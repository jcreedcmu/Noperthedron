## New file: `Noperthedron/Checker/Global.lean`

All checker code lives here. This file imports `SolutionTable.Basic` for
the `Row`/`Interval` types but does NOT import any `RationalApprox` or
`Global` modules. Everything is computable ‚ÄĒ no `noncomputable` keyword.

### Why a separate file?

The existing `sin‚Ąö`, `cos‚Ąö`, `rotM‚Ąö_mat`, etc. in
`RationalApprox/Basic.lean` are all inside a `noncomputable section`.
A `Bool`-returning checker cannot call noncomputable definitions. We
reimplement the same math as computable functions over ‚Ąö. This
duplication is intentional and acceptable ‚ÄĒ the checker is a
self-contained computational artifact, separate from the proof
infrastructure. This is sensible, because the function of `sin‚Ąö` is to
define the Taylor-series approximation of sin for the purposes of
*reasoning* about it. It is okay if we define a separate function that
takes rationals as inputs and outputs rationals for the purposes of
*computing* with it, and prove that the two are equivalent under
coercion.

#### B. Computable sinQ / cosQ

Reimplemented using the continued-fraction form from the Python
`helper_functions.py`, which is more efficient than `Finset.sum` and
manifestly computable:

```

12 nesting levels each, giving degree-25 (sin) and degree-24 (cos) polynomials.
Mathematically identical to `sin_psum 13` and `cos_psum 13` in

#### D. Rational vertex data

```

```

These are rational approximations of the 90 noperthedron vertices,
hard-coded from the Python's `RatNOP` computation (see "Vertex
approximation strategy" below). Each coordinate has the form
`‚Ćäexact ¬∑ 10¬Ļ‚Ā∂‚Ćč / 10¬Ļ‚Ā∂`, i.e. denominator 10¬Ļ‚Ā∂ with a ~17-digit
integer numerator.

The indexing convention matches the Python NOP matrix and the Lean
`nopertList`:

```

where k ‚ąą [0,15) is the rotation index, i ‚ąą {0,1,2} selects C1/C2/C3,

## Vertex approximation strategy

The real nopert vertices are defined as:

```
nopertPt k ‚Ąď i = (-1)^‚Ąď ¬∑ Rz(2ŌÄk/15)(CŠĶĘ)
```

where `Rz` uses exact `cos(2ŌÄk/15)` and `sin(2ŌÄk/15)` ‚ÄĒ irrational
values. Any rational approximation must deal with this irrationality,
and for M2 we must *prove* the approximation is within őļ = 10‚ĀĽ¬Ļ‚Āį.

### Chosen approach: Hard-code from Python (Approach A)

The Python does `‚Ćäexact ¬∑ 10¬Ļ‚Ā∂‚Ćč / 10¬Ļ‚Ā∂`, producing 270 rational numbers.
We generate these with a Python/Sage script and paste them into Lean as
a literal array. This is guaranteed to make the G > maxH inequality hold,
since it uses the exact same vertex data that the Python verification
already validated against.

### Rejected alternative: Compute algorithmically (Approach B)

We considered defining the rational vertices using our computable
`sinQ`/`cosQ` applied to rational angle approximations (e.g.
`cosQ(2 ¬∑ ŌÄQ ¬∑ k / 15)` where ŌÄQ is a rational approximation of ŌÄ).
This would avoid a data generation script and define vertices from first
principles.

We rejected this because it uses *different* rational vertices than
the Python, meaning the G and maxH values would be slightly different.
While the inequality should still hold (the G‚Ąö/H‚Ąö formulas include
őļ-correction terms designed to absorb vertex error up to őļ, and both
approximations are well within őļ), there could be razor-thin margins on
some rows. Using the Python's exact data eliminates this risk entirely.

### M2 proof of őļApproxPoly (both approaches need similar work)

For M2, we must prove `‚ÄĖreal_vertex - hard_coded_rational‚ÄĖ ‚Č§ őļ` for
each of the 90 vertices. This decomposes into:

1. **Taylor remainder bound:** `|sin(x) - sinQ(x)| ‚Č§ |x|¬≤‚Ā∑/27!` for
   `|x| ‚Č§ 4`. At x = 4 this is ~1.7 ¬∑ 10‚ĀĽ¬Ļ¬≤. The existing BoundsKappa
   module proves related bounds (e.g. `‚ÄĖrotM - rotM‚Ąö‚ÄĖ ‚Č§ őļ`), so the
   machinery exists or is close.

2. **ŌÄ approximation bound:** `|ŌÄ - ŌÄQ| ‚Č§ 10‚ĀĽ¬Ļ‚Ā∂` or similar. Mathlib
   has `Real.pi_lt_3141593` and `Real.pi_gt_3141592` which give ~7
   digits. We may need tighter bounds, but these are standard.

3. **Truncation bound:** The hard-coded value is `‚Ćäexact_rational ¬∑ 10¬Ļ‚Ā∂‚Ćč / 10¬Ļ‚Ā∂`
   where `exact_rational = cosQ(rational_angle) ¬∑ CŠĶĘ‚Īľ - sinQ(rational_angle) ¬∑ CŠĶĘ‚āĖ`
   (etc.). So the hard-coded value differs from the sinQ/cosQ-computed
   value by at most 10‚ĀĽ¬Ļ‚Ā∂ per coordinate. The overall proof chain is:

   ```
   real_vertex ‚Čą sinQ/cosQ_vertex ‚Čą hard_coded_vertex
        (Taylor+ŌÄ error ~10‚ĀĽ¬Ļ¬≤)    (truncation error ‚Č§ 10‚ĀĽ¬Ļ‚Ā∂)
   ```

   Total error ‚Č™ őļ = 10‚ĀĽ¬Ļ‚Āį.

4. **Composition:** The per-coordinate error of a vertex is bounded by
   `|CŠĶĘ‚Īľ| ¬∑ (trig_error)`, and since `‚ÄĖCŠĶĘ‚ÄĖ ‚Č§ 1`, the L2 norm error is
   at most `‚ąö3 ¬∑ max_coord_error ‚Čą ‚ąö3 ¬∑ 10‚ĀĽ¬Ļ¬≤ ‚Č™ őļ`.

## Relationship to M2

M1 produces a `Bool`-returning checker. M2 must prove:

```
checkGlobal row = true ‚Üí ¬¨ ‚ąÉ q ‚ąą row.toPoseInterval, RupertPose q nopert.hull
```

This requires three things, all M2 concerns:

1. **sinQ/cosQ agreement:** The computable `sinQ`/`cosQ` in
   `Checker/Global.lean` agree with the noncomputable `sin‚Ąö`/`cos‚Ąö` in
   `RationalApprox/Basic.lean` when evaluated at rational points (cast
   to ‚ĄĚ). This should be a straightforward proof that the
   continued-fraction form equals the Finset.sum form, since they compute
   the same polynomial.

2. **őļApproxPoly for vertices:** `nopertListQ` satisfies
   `őļApproxPoly nopertVerts nopertListQ_as_reals`. This requires the
   Taylor remainder + ŌÄ approximation bounds discussed in "Vertex
   approximation strategy" above. The existing BoundsKappa module proves
   similar bounds for rotation matrices and may provide reusable
   ingredients.

3. **Interval containment:** The axis-aligned box from
   `row.toPoseInterval` is contained in `pbar.closed_ball őĶ` where pbar
   is the center and őĶ is the max half-width. This is a simple geometric
   fact about sup-norm balls containing axis-aligned boxes.

## Performance notes

- Each `checkGlobal` call evaluates sinQ/cosQ (degree 25) at ~4 rational
  angles, computes ~10 matrix-vector products (2√ó3 or 2√ó2), and iterates
  over 90 vertices for maxHQ. This is ~1000 rational multiplications per row.

- For `#eval` (interpreted), one row should take well under a second.

- For `native_decide` on the full table (~17.5M global rows), performance
  will depend on the compiled code speed. This is an M5 concern.

- The continued-fraction form of sinQ/cosQ avoids computing large
  factorials and large powers separately, keeping intermediate rational
  numerator/denominator sizes more manageable.
